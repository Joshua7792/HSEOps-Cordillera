Three approaches — which fits best?

All three keep your launch scripts and PDF parsing logic untouched. The difference is how the database layer is built underneath.

A

Thin adapter — minimal change

Add a db_reader.py beside excel_reader.py. It returns the exact same data structures the frontend already expects, but reads from SQLite instead of .xlsx. The rest of the backend is untouched.

Pros

  • Least risk of breaking things
  • Frontend needs zero changes
  • Rollback is easy

Cons

  • Excel-shaped data model lives on in DB
  • Harder to add CRUD later
  • Duplicates some logic
B

SQLAlchemy models — clean DB layer ⭐ Recommended

Define proper ORM models: Worker, Contractor, Cert, CertEntry. Replace only excel_reader.py with DB queries. PDF import writes to these models. Launch scripts, API routes, and frontend stay the same.

Pros

  • Clean relational model — easy CRUD
  • Auto-migrations with Alembic
  • Edit records trivially
  • Same API responses — frontend unchanged

Cons

  • More upfront work than A
  • Adds SQLAlchemy dependency
C

Raw SQLite — no ORM

Use Python's built-in sqlite3 module directly. Write raw SQL queries everywhere. No new dependencies.

Pros

  • Zero new dependencies
  • Full control over every query

Cons

  • Lots of boilerplate SQL strings
  • Harder to maintain and extend
  • CRUD endpoints become verbose
  • No migration tooling

Why B is the right pick

You want full CRUD + import + warnings. SQLAlchemy gives you all of that with clean model definitions. The launch scripts stay identical — they just start uvicorn the same way they always have. Only excel_reader.py and workbook_sync.py get replaced.
